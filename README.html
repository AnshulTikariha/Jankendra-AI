Project overview

Local-first governance intelligence for leaders and staff

Jankendra-AI is designed as a local-first AI platform for constituency management. It brings together citizen complaints, meeting transcripts, ward data, commitments, development priorities, and historical facts so representatives can make faster and more transparent decisions.

Primary users

Leaders and staff members who need one place to monitor work, assign actions, and review ward-level insights.

Core goal

Turn scattered governance inputs into ranked to-do items, commitment tracking, development priorities, and RAG-backed answers.

Deployment approach

Start on one local machine with SQLite, in-process AI modules, and local production LLM support before scaling to SaaS.

Product capabilities

What the platform is planned to deliver

Complaint clustering

Similar ward-level complaints are grouped with embeddings so repeated citizen issues become visible as high-priority clusters.

  • Ward scoped
  • Append-only audit
  • Department routing

Commitment tracking

Meeting transcripts can be processed into commitments with deadlines, escalation weights, and completion history.

  • Transcript upload
  • Deadline weights
  • Resolved archive

Development prioritization

Ward data, open complaint clusters, infrastructure needs, and scheme penetration inform a ranked development plan.

  • Multi-factor score
  • Reasoning trace
  • Ward comparison

RAG assistant

A grounded assistant answers constituency questions using profile data, digest metrics, historical facts, SQL results, and chat history.

  • Hybrid retrieval
  • Cited answers
  • SSE streaming

Weekly digest

SQL-first digest generation highlights open complaints, overdue work, resolved commitments, and measurable constituency changes.

  • No LLM required
  • Scheduled job
  • Drilldown ready

Context injection

Staff can add trusted text documents to the knowledge base so future answers reflect local priorities and latest facts.

  • .txt upload
  • Knowledge base
  • Staff controlled
Architecture

Three layers with clear boundaries

1

Frontend layer

React, TypeScript, and Tailwind provide the dashboard, to-do list, commitments, chat, digest, issue logging, and staff workflows.

2

Backend layer

FastAPI acts as the security boundary, API gateway, orchestration layer, job runner, and system of record.

3

AI modules layer

Independent Python engines handle ingestion, clustering, commitment extraction, prioritization, RAG, and digest logic.

Frontend React dashboard, forms, chat, digest, and staff tools
Backend FastAPI, JWT/RBAC, repositories, services, background jobs
AI modules Embeddings, LLM providers, vector search, ranking engines
Data model

Four logical stores inside one physical database

Store Purpose Write policy Example tables
DB1 - Static Constituency Ward demographics, infrastructure, schemes, and constituency profile data. Staff only. wards, demographics, infrastructure, schemes
DB2 - RAG Historical Facts Meeting summaries, resolved commitments, and historical cluster snapshots. System jobs and staff actions. meeting_summaries, resolved_commitments, cluster_snapshots
DB3 - Timely Commitments Active commitments, deadlines, weights, escalation status, and history. System ingestion and staff actions. commitments, commitment_history
DB4 - Immutable Complaints Raw complaint audit trail and complaint clusters for issue intelligence. Append-only system writes. complaints, complaint_clusters
Technology stack

Modern, typed, and portable from development to production

Frontend

React 18+, TypeScript, Tailwind CSS, Vite, TanStack Query, Zustand, OpenAPI-generated API client.

Backend

Python 3.11, FastAPI, Uvicorn, SQLAlchemy 2.0, Alembic, Pydantic v2, JWT with RBAC.

AI and retrieval

sentence-transformers, Gemini for development, Ollama and Sarvam for local production, explicit pipelines.

Database

SQLite with sqlite-vec in development and PostgreSQL 16+ with pgvector in production.

Security and privacy

Privacy is enforced by the architecture

Local-first production mode

Audio recordings, transcripts, constituency data, complaints, commitments, and AI responses are designed to stay on the local machine in production.

Controlled data writes

Static constituency data is staff-write-only, complaints are append-only, and AI modules return typed results without receiving raw database access.

Role-based access

JWT claims separate leader and staff permissions across dashboard, data management, and action workflows.

Provider abstraction

Development can use Gemini while production switches to local Ollama and Sarvam through configuration instead of application rewrites.

Roadmap

Planned delivery phases

Phase 0 - Foundation

Monorepo scaffold, configuration, OpenAPI, provider interfaces, migrations, and documentation.

Phase 1 - Core backend

Database schema, seed data, authentication, constituency APIs, dashboard API, and frontend layout.

Phase 2 - Complaint intelligence

Embeddings, sqlite-vec, complaint clustering, department routing, and weighted to-do list.

Phase 3 - Commitments

Transcript parsing, commitment extraction, upload jobs, escalation, and resolved archives.

Phase 4 - Priorities and digest

Development ranking, ward comparison, SQL digest aggregation, and digest dashboard views.

Phase 5 and 6 - RAG and hardening

Grounded assistant, context injection, local LLM support, tests, accessibility, performance, and Hindi foundation.
